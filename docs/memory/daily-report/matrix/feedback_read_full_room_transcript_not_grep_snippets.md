Grabbing 2-3 lines via keyword grep (e.g. searching for the project name) from a large active room and calling it "clean, no action needed" is not sufficient — it misses real content sitting a few lines away.

**Why:** 2026-07-07, "Arthur - Meta-Stamp" room had 79 messages. A quick grep pulled only "namtv: asking about >40h/week policy... client budget concerns" and this was filed as "informational, internal team discussion, no action needed." Reading the FULL 79-message transcript revealed: (1) a real unresolved client question about metadata/attribution requirements nobody had answered yet, (2) a demo pushed to Production for client testing the next day, (3) and — the actually important part — the team lead (Nam Tran) explicitly saying **"lately several clients have been difficult about overtime pay, one even refused to pay"** — a real pattern of client payment disputes, not routine internal chatter. User: "Sao ko thấy report của Arthur mà lại mark done, thấy quá trời info kìa!!!" (why wasn't there a report when there's clearly tons of info).

**How to apply:**
- For any room with double-digit+ message counts in a monitoring window, or any room tied to an active/named client project, read the room's FULL transcript (not just a grep hit) before writing the daily-report summary or deciding a Trello item is "clean."
- Watch specifically for: unanswered client questions, financial/payment friction phrases (unpaid overtime, budget complaints, "khó", "không thanh toán"), and deliverable deadlines (production pushes, demo dates) — these get missed by narrow keyword searches because they don't contain the searched keyword itself.
- See [[feedback_matrix_daily_summary]] for the report FORMAT (bullets not raw dump) — this memory is about READ DEPTH before summarizing, a different failure mode than formatting.

🔴 **RECURRED 2026-07-14, different sub-mistake — computing a timing estimate from ONE isolated quote instead of cross-referencing all timing-related messages.** Arthur/Meta-Stamp: read only a handful of recent messages, found "My demo with Chris and Casey is tomorrow at 7pm my time" (Art, Mexico timezone) and concluded THE real investor demo = ~08:00 VN next day. User pushed back TWICE ("check carefully", "dịch hết msg qua xem, có gì đó sai sai") before a full 216-message re-read of the actual Slack channel (not just Matrix) revealed **two separate events**: (1) that "7pm my time" call was an internal Art-Chris-Casey prep/review call, and (2) a SEPARATE sentence a few messages later — "I have one [call] with him before the start of your day tomorrow, and he has the real one shortly after the end of that same day" — placed the ACTUAL investor demo many hours earlier than my estimate (same-day evening, not next-morning).
- **Escalated rule:** when a room/channel discusses an event with a client-side scheduling deadline (demo, launch, freeze), grep for EVERY message mentioning that event/time (not just the first one found) before computing a single derived estimate. A single quoted time from one person in one timezone is not the full picture if the same conversation contains a second, more specific breakdown ("X happens at A, Y happens at B") — read forward and backward from the first match, not just forward from where you started.
- Also: this was a case where the *deeper* data source (full Slack `conversations.history`, 216 messages) contained the disambiguating context that a shallower check (Matrix rooms only, or a handful of recent Slack messages) did not — when a timing/outcome question is unresolved and important, prefer pulling complete history over recent-N-messages sampling.
