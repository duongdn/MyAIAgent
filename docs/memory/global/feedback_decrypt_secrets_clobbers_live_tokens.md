**Rule:** Don't reflexively run `bash scripts/decrypt-secrets.sh` at session start out of habit ([[feedback_decrypt_before_reading]]) without considering it can destroy a token refreshed more recently than the last `.enc` commit. Token-refresh scripts previously only wrote the plaintext `config/*.json`, never re-encrypted back to `.enc` â€” so any later `decrypt-secrets.sh` reverted to whatever was last committed, forcing an unnecessary re-auth.

**Why:** 2026-07-08 â€” Matrix token was refreshed and working fine at 08:54 and 10:01 (confirmed via real Matrix room fetches). Ran `decrypt-secrets.sh` reflexively at session start ~10:51, which overwrote the live token with the `.enc` committed 2026-07-07 10:36 â€” instantly breaking a working session and forcing a full re-login. User was rightly frustrated ("dĂąng cáşŁ ngĂ n láş§n lĂ˘u nay rá»“i sao giá»ť láşˇi lá»—i") since this wasn't a real Matrix/session problem â€” it was self-inflicted by the decrypt call.

**Fix applied:** created `scripts/lib/save-secret-config.js` (`saveSecretConfig(path, config)` â€” writes JSON then immediately runs `encrypt-secrets.sh`). Wired into all 3 Matrix token-refresh scripts' save points (`matrix-token-refresh.js`, `matrix-device-auth.js`, `matrix-token-cdp-refresh.js`). Any future refreshed token is now immediately persisted to `.enc`.

**How to apply:** If any other script (Slack xoxc, Workstream login, etc.) writes a decrypted config in place after a refresh, wire it through the same `saveSecretConfig` helper, not raw `fs.writeFileSync`. Before assuming "token expired" when a call fails right after a `decrypt-secrets.sh` run, check whether the decrypt itself just clobbered a live token â€” verify via `whoami`/equivalent before jumping to a re-auth flow.

**2nd occurrence, worse variant, 2026-07-13 (Arthur Piece 13 daily run):** this time it wasn't a revert-to-stale, it was a full **deletion**. The "Solid Code Team" Slack account entry (`config/.slack-accounts.json`, added 2026-07-06/07 per `feedback_solid_code_new_workspace_unwired`) had been added directly to the plaintext JSON and **never once re-encrypted into `.enc`** (it predates `saveSecretConfig` and was never migrated). Sometime the morning of 2026-07-13 a `decrypt-secrets.sh` run overwrote `config/.slack-accounts.json` with the old `.enc` (which never contained Solid Code at all) â€” the account didn't revert to stale, it vanished from the account list entirely. Confirmed via `.json` mtime (08:59) being newer than `.enc` mtime (08:34) that same morning, and `git log` on the file being empty (gitignored, no history to diff against). Recovery attempt: re-extracted David's live cookies + navigated Chrome Profile 15 directly to `solid-code-team.slack.com` â€” session was still alive/logged-in (confirmed visually), but the cached xoxc token in that profile's local storage was stale (Feb-06, `invalid_auth`) and getting a fresh one required more live-desktop interaction than was safe to do while the user was actively on the same shared display (see `feedback_gui_automation_risk_on_shared_desktop`). Left unresolved â€” token needs re-extraction next time the desktop is free.

**Broader lesson:** this bug class isn't limited to Matrix. ANY secret added directly to a plaintext `config/*.json` without going through `saveSecretConfig` (or a manual `encrypt-secrets.sh` right after) is one `decrypt-secrets.sh` run away from silent, total loss â€” not just staleness. Audit: check whether `config/.slack-accounts.json`'s other manually-added session accounts (Amazing Meds, Equanimity, OhCleo) are safely mirrored into `.enc` â€” if any were hand-edited post-refresh without re-encrypting, they have the same exposure.
